# /tune-meeting-notes — 회의록 미세조정

> 사용 시점: W5 양식 lock 후, 받은 회의록이 *어딘가 아쉬울 때* (선택).
> `/tune-persona` 의 회의록 버전이에요 — 받은 결과를 paste 받아 1~2줄 처방.

## 갱이가 할 일

### 1. 사용자에게 받은 회의록 paste 받기

> "최근에 받은 회의록 중 *아쉬웠던 것* 을 붙여 넣어 주세요. 어디가 아쉬웠는지도 한 줄로요 🐾
>  (붙여 넣은 건 갱이가 진단용으로만 봐요 — 어디 저장 안 해요.)"

### 2. 증상별 처방 — 가장 흔한 4가지

**증상 A — 사실이 틀어졌다 (이름·날짜·기한이 바뀜)** ← *가장 중요*

- 원인: 페르소나 시스템 프롬프트가 너무 *자유분방* 하거나, 페르소나 본체에 "각색을 좋아함" 같은 톤이 있음.
- 처방: `persona/system_prompt.md` 에 한 줄 추가 — "사실(이름·숫자·날짜)은 입력 그대로 옮긴다."
  - 회의록 지침(코드)에도 사실 보존이 박혀 있지만, *페르소나 본체가 강하게 각색형*이면 본체 한 줄이 더 효과적.

**증상 B — 너무 길다 / 너무 짧다**

- 처방: `/setup-meeting-style` 의 length 를 short ↔ medium ↔ long 으로 한 칸 조정.

**증상 C — 페르소나 톤이 너무 과해서 회의록 가독성이 떨어진다**

- 원인: 페르소나가 매 줄에 감탄사·이모지를 박음.
- 처방: yml 은 톤을 직접 못 줄여요. `persona/system_prompt.md` 에 "문서·회의록을 쓸 땐 톤을 절제한다" 한 줄.
  - 또는 페르소나를 살짝 차분한 톤으로 `/tune-persona`.

**증상 D — 액션아이템에 담당자/기한이 자꾸 '(미정)'**

- 원인: *원본 메모에 담당/기한이 안 적혀 있어서* — 코드가 의도적으로 지어내지 않아요 (사실 보존).
- 처방: 이건 *정상 동작*. 회의 때 담당·기한을 메모에 적는 습관을 들이면 회의록 품질이 올라가요. (도구가 아니라 입력의 문제.)

### 3. 처방 적용

증상에 따라:
- yml 조정 → `/setup-meeting-style` 안내 (재시작 불필요)
- 페르소나 한 줄 → `persona/system_prompt.md` 편집 후 *봇 재시작* (페르소나는 부팅 시 1회 로드)

```bash
# 페르소나 수정 후 적용
./scripts/stop.sh && ./scripts/start.sh
```

### 4. **여기까지 됐나요?** — 페이스 멈춤

> "처방 적용했으면 같은 메모로 한 번 더 굴려서 나아졌는지 보세요. 사실(이름·날짜)이 맞고 양식이 마음에 들면 그게 결승선이에요 🐾"

## 절대 안 할 것

- 사용자의 회의 *본문* 을 어디에 저장하거나 다른 곳에 보내지 않기 — 진단용으로만 보고 흘려보내기.
- 사실 보존을 약화하는 처방(예: "더 자유롭게 각색하게 하자") 하지 않기 — 회의록의 생명은 사실.
- 페르소나 본체를 사용자 승인 없이 임의 수정하지 않기 — 제안까지, 수정은 사용자 손.

## FAQ

**Q. 같은 메모인데 굴릴 때마다 회의록이 조금씩 달라요.**
A. 정상이에요 — LLM 특성이에요. *사실(이름·숫자·날짜)* 만 일정하면 OK. 표현이 매번 똑같길 원하면 length 를 short 로 줄이면 변동이 작아져요.

**Q. 회의록을 노션에 그대로 붙였더니 체크박스가 깨져요.**
A. yml `action_item_format` 을 `checkbox` 로 두면 `- [ ]` 형식이라 노션·텔레그램에서 체크리스트로 인식돼요. 깨지면 numbered 로 바꿔 보세요.
